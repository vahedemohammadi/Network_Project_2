### تحلیل هدرهای HTTP و پشته پروتکلی TCP/IP

# هدف

هدف از فاز دوم، تحلیل هدرهای پروتکل یک بسته HTTP ضبط‌شده با استفاده از نرم‌افزار **Wireshark** است. در این مرحله، بسته شبکه در سه لایه اصلی پشته پروتکل **TCP/IP** شامل لایه کاربرد (Application)، لایه انتقال (Transport) و لایه شبکه (Network) مورد بررسی و تحلیل قرار می‌گیرد.

---

# فیلتر نمایش (Display Filter)

برای پیدا کردن بسته‌های HTTP در نرم‌افزار Wireshark از فیلتر زیر استفاده شد:

```text
http
```

همچنین می‌توان از فیلتر زیر نیز برای نمایش بسته‌های مربوط به پروتکل HTTP استفاده کرد:

```text
tcp.port == 80
```

---

# بسته انتخاب‌شده

برای انجام تحلیل، اولین درخواست **HTTP GET** که توسط ابزار **cURL** ایجاد شده بود انتخاب شد. این بسته، آغازگر ارتباط میان کلاینت و سرور بوده و اطلاعات مهمی از هدرهای پروتکل را در اختیار قرار می‌دهد.

---

# تحلیل لایه کاربرد (Application Layer - HTTP)

## اطلاعات استخراج‌شده

| فیلد | مقدار |
|------|--------|
| روش درخواست (HTTP Method) | GET |
| نسخه HTTP | HTTP/1.1 |
| میزبان (Host) | neverssl.com |
| عامل کاربر (User-Agent) | curl/8.13.0 |

## تحلیل

بسته انتخاب‌شده شامل یک درخواست **HTTP GET** است که توسط نرم‌افزار **cURL** ارسال شده است. این درخواست از نسخه **HTTP/1.1** استفاده می‌کند و در هدر **Host** نام دامنه مقصد (**neverssl.com**) مشخص شده است. همچنین هدر **User-Agent** نشان می‌دهد که درخواست توسط نرم‌افزار **cURL نسخه 8.13.0** تولید شده است.

روش **GET** برای دریافت اطلاعات از سرور استفاده می‌شود و برخلاف روش‌هایی مانند POST، تغییری در منابع موجود روی سرور ایجاد نمی‌کند. در این درخواست، کلاینت تنها تقاضای دریافت محتوای صفحه موردنظر را به سرور ارسال کرده است.

---

# تحلیل لایه انتقال (Transport Layer - TCP)

## اطلاعات استخراج‌شده

| فیلد | مقدار |
|------|--------|
| پروتکل | TCP |
| پورت مبدأ | 10840 |
| پورت مقصد | 80 |

## تحلیل

در این ارتباط از پروتکل **TCP (Transmission Control Protocol)** استفاده شده است. پورت مبدأ (**10840**) یک پورت موقت (Ephemeral Port) است که به‌صورت خودکار توسط سیستم‌عامل برای این ارتباط اختصاص داده شده است. پورت مقصد (**80**) نیز پورت استاندارد سرویس **HTTP** محسوب می‌شود.

پروتکل TCP وظیفه برقراری یک ارتباط مطمئن میان کلاینت و سرور را بر عهده دارد. این پروتکل با استفاده از شماره‌گذاری بسته‌ها، کنترل خطا، بررسی صحت داده‌ها و ارسال مجدد بسته‌های ازدست‌رفته، اطمینان حاصل می‌کند که اطلاعات به‌صورت صحیح، کامل و به همان ترتیبی که ارسال شده‌اند به مقصد برسند.

---

# تحلیل لایه شبکه (Network Layer - IPv4)

## اطلاعات استخراج‌شده

| فیلد | مقدار |
|------|--------|
| آدرس IP مبدأ | 192.168.1.12 |
| آدرس IP مقصد | 34.223.124.45 |

## تحلیل

آدرس IP مبدأ (**192.168.1.12**) مربوط به رایانه کلاینت در شبکه محلی است. آدرس IP مقصد (**34.223.124.45**) نیز متعلق به سرور وب میزبان دامنه **neverssl.com** است.

در این مرحله، پروتکل **IPv4** مسئول آدرس‌دهی و مسیریابی بسته‌ها در سطح شبکه است. بسته‌های داده با استفاده از این آدرس‌ها از طریق تجهیزات شبکه مانند روترها به سمت مقصد هدایت می‌شوند تا در نهایت به سرور موردنظر برسند.

---

# فرآیند کپسوله‌سازی (Packet Encapsulation)

بسته ضبط‌شده، فرآیند کپسوله‌سازی اطلاعات در پشته پروتکل TCP/IP را به‌خوبی نمایش می‌دهد. این فرآیند به ترتیب زیر انجام می‌شود:

1. درخواست **HTTP** در لایه کاربرد ایجاد می‌شود.
2. درخواست HTTP داخل یک سگمنت **TCP** قرار می‌گیرد.
3. سگمنت TCP داخل یک بسته **IPv4** کپسوله می‌شود.
4. بسته IP از طریق شبکه به سمت سرور مقصد ارسال می‌شود.

این ساختار نشان می‌دهد که هر لایه، اطلاعات لایه بالاتر را به‌عنوان داده دریافت کرده و هدر مخصوص خود را به آن اضافه می‌کند تا بسته برای انتقال در شبکه آماده شود.

# نتیجه‌گیری

در این فاز، بسته مربوط به درخواست **HTTP GET** با موفقیت توسط نرم‌افزار **Wireshark** مورد بررسی قرار گرفت. اطلاعات مهم موجود در هدرهای لایه کاربرد، لایه انتقال و لایه شبکه استخراج و تحلیل شدند و نحوه عملکرد هر یک از این لایه‌ها در فرآیند ارسال درخواست به سرور بررسی گردید.

همچنین فرآیند کپسوله‌سازی داده‌ها در پشته پروتکل **TCP/IP** به‌صورت عملی مشاهده شد و مشخص گردید که چگونه یک درخواست HTTP پس از عبور از لایه‌های مختلف شبکه، به سرور مقصد منتقل می‌شود. انجام این مرحله باعث درک بهتر ساختار بسته‌های شبکه، نحوه عملکرد پروتکل‌های ارتباطی و ارتباط میان لایه‌های مختلف مدل TCP/IP گردید.
